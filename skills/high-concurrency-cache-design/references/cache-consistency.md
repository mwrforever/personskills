# 缓存一致性方案全景矩阵

## 核心原则

**"没有银弹，只有权衡"**。不存在完美的缓存一致性方案，每种方案都有其适用场景和代价。

---

## 方案 A：先删除缓存，再更新数据库

### 流程

```
1. 删除缓存
2. 更新数据库
```

### 异常分析

```
时刻 T1: 线程A 删除缓存
时刻 T2: 线程B 读取 -> 缓存不存在 -> 读DB旧值 -> 写回缓存
时刻 T3: 线程A 更新数据库完成
结果：缓存中是旧值，数据库是新值 → 永久脏数据
```

### 适用场景

**几乎不适用**。除非：
- 并发极低（< 10 QPS）
- 对脏读的容忍度极高（无所谓）
- 有其他机制兜底（如：定期全量刷新）

---

## 方案 B：先更新数据库，再删除缓存（最常用标配）

### 流程

```
1. 更新数据库
2. 删除缓存
```

### 异常分析

```
时刻 T1: 缓存刚好失效
时刻 T2: 线程A 读取 -> 读DB旧值
时刻 T3: 线程B 更新DB -> 删除缓存
时刻 T4: 线程A 把旧值写回缓存
结果：产生脏数据
```

### 发生概率分析

| 条件 | 概率影响 |
|------|----------|
| 数据库写操作比读慢 | 大多数场景下成立 |
| 线程A 读操作比线程B 写操作快 | 小概率 |
| 线程A 在写回缓存前，线程B 已经删除了缓存 | 需要刚好在这个窗口 |

**结论**：由于数据库写操作通常比读慢，且缓存删除比读更快，这种时序发生的概率极低（< 0.1%）。

### 适用场景

绝大部分"读多写少"的常规业务：
- 商品信息
- 用户资料
- 配置项
- 分类/标签

---

## 方案 C：延时双删（方案 B 的补丁）

### 流程

```java
// 线程A 更新操作
1. 删除缓存
2. 更新数据库
3. Thread.sleep(N 毫秒) // N = 预估的业务执行时间
4. 再次删除缓存
```

### 异常分析

- Sleep 时间难以精准评估（业务逻辑可能变化）
- 会导致接口 RT 波动
- 第二轮删除失败依然会产生脏数据（需配合 MQ 重试）

### 适用场景

对一致性要求稍高，但不想引入 Canal 等中间件的过渡方案。

### 改进版：异步双删

```
1. 删除缓存
2. 更新数据库
3. 发送消息到 MQ
4. 消费者延迟 N 秒后删除缓存（避免 Sleep 阻塞主线程）
```

---

## 方案 D：基于 Binlog 异步订阅（Canal/MQ，最终一致性终极方案）

### 流程

```
1. 业务代码只管更新数据库
2. Canal 订阅 MySQL Binlog
3. 解析 Binlog，发送变更消息到 MQ
4. 消费者从 MQ 消费消息，删除对应缓存
```

### 优点

| 优点 | 说明 |
|------|------|
| 代码侵入性低 | 业务代码只需关注 DB 操作 |
| 异步化 | 不影响主流程性能 |
| 最终一致性 | 延迟通常在毫秒级 |

### 缺点

| 缺点 | 说明 |
|------|------|
| 引入中间件 | 需要部署 Canal + MQ |
| 复杂性增加 | 需要维护额外组件 |
| 延迟 | 可能存在几秒延迟 |

### 适用场景

高并发、微服务架构、对代码侵入性要求极低的中大型互联网标配。

---

## 方案 E：强一致性（分布式事务/读写锁）

### 适用场景

资金、库存等**绝对不允许脏读**的场景。

### 实现方式

#### 方式 1：分布式事务（Seata / ShardingSphere SphereEx）

```java
@GlobalTransactional
public void updateInventory(int goodsId, int count) {
    // 1. 更新数据库
    inventoryMapper.update(goodsId, count);
    // 2. 删除缓存（同一事务内）
    redisTemplate.delete("inventory:" + goodsId);
}
```

**代价**：吞吐量断崖式下跌（可能降低 10 倍以上）。

#### 方式 2：Redisson 读写锁串行化

```java
RReadWriteLock rwLock = redissonClient.getReadWriteLock("inventory:lock:" + goodsId);
RLock writeLock = rwLock.writeLock();
writeLock.lock(); // 所有读写操作都必须排队
try {
    // 1. 读DB
    // 2. 写DB
    // 3. 删除缓存
} finally {
    writeLock.unlock();
}
```

**代价**：完全串行化，高并发下性能极差。

## 方案对比矩阵

| 方案 | 一致性强度 | 性能影响 | 实现复杂度 | 适用场景 |
|------|------------|----------|------------|----------|
| A：先删缓存再更DB | 弱（可能永久脏数据） | 低 | 低 | 几乎不用 |
| B：先更DB再删缓存 | 弱（低概率脏数据） | 低 | 低 | 读多写少常规业务 |
| C：延时双删 | 中等（大幅降低脏数据概率） | 中（RT 波动） | 中 | 过渡方案 |
| D：Binlog 订阅 | 最终一致（毫秒级） | 低 | 高 | 大型互联网标配 |
| E：分布式事务/锁 | 强一致 | 极高（10x+ 下降） | 高 | 资金/库存等关键场景 |

---

## 选择决策树

```
对脏读的容忍度？
├── 容忍（无所谓） → 方案 B（先更新DB再删缓存）
├── 极低（不能接受脏读） → 下一步
└── 需要精确计数 → 下一步

是否高并发场景？
├── 是 → 方案 D（Binlog 订阅）
└── 否 → 方案 E（分布式事务/锁）

是否资金/库存等关键场景？
├── 是 → 方案 E（强一致）
└── 否 → 方案 D 或 方案 B
```

## 异常兜底机制

无论选择哪种方案，都需要考虑以下异常情况：

### 1. 缓存删除失败

```java
try {
    redisTemplate.delete(cacheKey);
} catch (Exception e) {
    // 记录日志，发送告警
    // 写入重试队列，稍后重试
    mqProducer.send("cache:retry:delete", cacheKey);
}
```

### 2. 缓存穿透兜底

无论哪种一致性方案，都应该在查询层加入穿透防护：

```java
// 布隆过滤器前置拦截
if (!bloomFilter.contains(key)) {
    return null; // 一定不存在
}

// 空值缓存防止穿透
String value = redisTemplate.opsForValue().get(key);
if (value == null) {
    // 从 DB 加载
    value = db.get(key);
    if (value != null) {
        redisTemplate.opsForValue().set(key, value, 5, TimeUnit.MINUTES);
    }
}
```

### 3. 缓存雪崩兜底

- 过期时间添加随机偏移（`actualTTL = baseTTL + random(0, baseTTL * 0.1)`）
- 多级缓存（本地 + 分布式）
- 热点数据永不过期（手动管理）
